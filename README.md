# TennisScoreboard

TennisScoreboard — учебное клиент-серверное Java web-приложение для ведения теннисных матчей.
Проект позволяет создавать новый матч, считать очки в реальном времени и просматривать список завершённых матчей с поиском и пагинацией.

Основная цель проекта — получить практику в классическом Java web-стеке без Spring Boot: **Servlet + JSP + Hibernate + H2**, а также закрепить паттерн **MVC(S)**, ручную вёрстку и базовую серверную архитектуру.

---

## Возможности приложения

### Работа с матчами

- создание нового матча;
- автоматическое создание игроков, если их ещё нет в БД;
- хранение **текущих матчей в памяти приложения**;
- подсчёт очков, геймов и сетов во время матча;
- завершение матча с сохранением результата в БД;
- просмотр списка завершённых матчей;
- поиск завершённых матчей по имени игрока;
- постраничный вывод результатов;
- обработка пользовательских ошибок и отдельных 404 / 500 страниц.

### Правила подсчёта счёта

Для упрощения используется следующая модель теннисного матча:

- матч играется в формате **best of 3** — до двух выигранных сетов;
- стандартный счёт внутри гейма: `0 → 15 → 30 → 40`;
- при `40:40` включается логика **deuce / advantage**;
- сет выигрывается при счёте `6:X`, если у соперника меньше 5 геймов;
- при `6:5` сет ещё не заканчивается — нужна разница в 2 гейма;
- при `6:6` запускается **тай-брейк**;
- тай-брейк играется до 7 очков, но также с преимуществом минимум в 2 очка.

## Архитектура

Проект построен вокруг паттерна **MVC(S)**:

- **Model** — доменная модель матча, счёта, игроков, DTO и сущности Hibernate;
- **View** — JSP-страницы;
- **Controller** — `HttpServlet`, принимающие GET/POST-запросы;
- **Service** — координация сценариев приложения: создание матча, обновление счёта, сохранение завершённых матчей.

### Что делает каждый слой

- **`web.servlet`** — принимает HTTP-запросы и решает, что показать пользователю;
- **`service`** — содержит прикладные сценарии и orchestration-логику;
- **`domain`** — хранит правила теннисного счёта;
- **`repository`** — контракты доступа к данным;
- **`storage.db.hibernate`** — Hibernate-реализация работы с БД;
- **`storage.inmemory`** — хранение активных матчей в памяти;
- **`filter`** — транзакции на запрос и логирование ошибок;
- **`web.bootstrap`** — инициализация инфраструктуры при старте приложения.

### Инициализация приложения

При запуске `AppContextListener`:

1. собирает Hibernate `Configuration`;
2. регистрирует entity-классы;
3. запускает Flyway-миграции;
4. создаёт `SessionFactory`;
5. создаёт repository/service-объекты;
6. кладёт зависимости в `ServletContext`.

Это даёт простой и понятный bootstrapping без DI-контейнера.

## Основные страницы и маршруты

### `GET /`
Главная страница.

Содержит ссылки на:
- создание нового матча;
- просмотр завершённых матчей.

### `GET /new-match`
Страница создания нового матча.

### `POST /new-match`
Создаёт матч:
- валидирует имена игроков;
- проверяет, существуют ли игроки в БД;
- при необходимости создаёт новых игроков;
- создаёт новый `Match`;
- сохраняет его в in-memory хранилище;
- делает redirect на страницу счёта.

### `GET /match-score?uuid={uuid}`
Показывает текущий счёт матча по UUID.

### `POST /match-score?uuid={uuid}`
Принимает ID игрока, выигравшего очко:
- обновляет доменный объект `Match`;
- если матч продолжается — делает redirect обратно на страницу счёта;
- если матч завершён — удаляет его из in-memory хранилища, сохраняет результат в БД и рендерит финальный счёт.

### `GET /matches?page={n}&name={playerName}`
Страница завершённых матчей.

Поддерживает:
- пагинацию;
- фильтр по имени игрока;
- отображение победителя;
- окно страниц для постраничной навигации.

## Хранение данных

### 1. Текущие матчи

Активные матчи хранятся **в памяти приложения**:

```text
ConcurrentHashMap<UUID, Match>
```

- ключ — `UUID` матча;
- значение — доменный объект `Match` с текущим состоянием счёта.

### 2. Завершённые матчи

После окончания матч сохраняется в **H2 in-memory database**.

Используются две таблицы:

#### `players`
- `id` — первичный ключ;
- `name` — уникальное имя игрока.

#### `matches`
- `id` — первичный ключ;
- `player1` — FK на `players.id`;
- `player2` — FK на `players.id`;
- `winner` — FK на `players.id`.

### SQL-миграции

База создаётся через **Flyway**:

- `V1__init_tables.sql` — схема таблиц и индексы;
- `V2__init_data.sql` — начальные данные для игроков и завершённых матчей.

## Тесты

Юнит-тестами покрыта доменная логика подсчёта счёта:

- обычный рост очков внутри гейма;
- победа в гейме;
- логика `deuce / advantage`;
- завершение сета;
- завершение матча;
- запуск и завершение tie-break;
- сброс очков и геймов после конца сета;
- исключения для невалидных сценариев.

Основные тестовые классы:

- `MatchTest`
- `MatchTieBreakTest`
- `MatchExceptionsTest`

## Как запустить локально

### Что понадобится

- **JDK 21**
- **Maven 3.9+** или Maven Wrapper
- **Apache Tomcat 11**

### 1. Клонировать проект

```bash
git clone git clone https://github.com/prplhd/TennisScoreboard.git
cd TennisScoreboard
```

### 2. Собрать WAR

#### Через Maven Wrapper

Linux / macOS:

```bash
chmod +x mvnw
./mvnw clean package
```

Windows:

```bat
mvnw.cmd clean package
```

#### Или через установленный Maven

```bash
mvn clean package
```

После сборки будет создан файл:

```text
target/TennisScoreboard.war
```

### 3. Задеплоить WAR в Tomcat

Скопируйте `TennisScoreboard.war` в папку `webapps` вашего Tomcat.

Пример:

```text
<TOMCAT_HOME>/webapps/TennisScoreboard.war
```

После запуска Tomcat приложение будет доступно по адресу:

```text
http://localhost:8080/TennisScoreboard/
```

### 4. Переменные окружения (опционально)

В `HibernateSessionFactoryProvider` предусмотрен override учётных данных БД через переменные окружения:

- `DB_USER`
- `DB_PASSWORD`

Для стандартной H2-конфигурации локально это обычно не требуется.

## H2 Console

В проекте подключена H2 web console.

Адрес:

```text
http://localhost:8080/TennisScoreboard/h2-console/
```

Параметры подключения:

```text
JDBC URL: jdbc:h2:mem:tennis;DB_CLOSE_DELAY=-1
User Name: sa
Password: [пусто]
```

В текущей конфигурации доступ к H2 Console разрешён удалённо (`webAllowOthers=true`), то есть консоль доступна не только локально.  
Это удобно для отладки, но небезопасно для production-окружения.
Если нужно закрыть внешний доступ, измените конфигурацию сервлета H2 в `web.xml` и отключите параметр, разрешающий remote access.
